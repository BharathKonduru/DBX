Foundations
Q 01
What is Delta Lake, and what core problem does it solve over standard data lakes?
Answer: B

Standard data lakes store files (Parquet, JSON, CSV) on object storage but suffer from: no atomicity (partial writes leave corrupt state), no isolation (concurrent reads/writes produce dirty reads), no schema enforcement (bad data silently corrupts datasets), and no easy rollback.

Delta Lake solves this by adding a transaction log (_delta_log/) on top of Parquet files. Every read and write goes through this log, giving the lake ACID guarantees previously only available in relational databases.

Key features:
• ACID transactions — atomic commits, serializable isolation
• Schema enforcement — rejects writes that don't match the table schema
• Schema evolution — controlled schema changes with audit trail
• Time travel — query historical versions
• Scalable metadata — handles tables with millions of files

Delta Lake is open-source (Apache 2.0 license) and works with Spark, Flink, Hive, and other engines.
Q 02
What is a Data Lake, and how does it differ from a Data Warehouse?
Answer: B

Data Lake:
• Stores ANY format: CSV, JSON, Parquet, images, logs, binary
• Schema-on-read: structure applied when data is queried
• Cheap object storage (S3, ADLS, GCS)
• High flexibility, lower governance
• Risk of becoming a "data swamp" without proper management

Data Warehouse:
• Stores structured, processed data only
• Schema-on-write: structure enforced at load time
• Optimized for OLAP queries (Redshift, Snowflake, BigQuery)
• High governance, lower flexibility
• Expensive for very large raw datasets

Modern trend — Lakehouse: Delta Lake + Spark combines both: raw storage cost of a lake + ACID + schema enforcement + performance of a warehouse. This is the "lakehouse" architecture.
Q 03
How is the Delta Lake transaction log structured and when are new log entries created?
Answer: B

The _delta_log/ directory lives alongside your Parquet data files. Structure:

my_table/
├── _delta_log/
│ ├── 00000000000000000000.json ← CREATE TABLE
│ ├── 00000000000000000001.json ← first INSERT
│ ├── 00000000000000000002.json ← UPDATE
│ ├── ...
│ ├── 00000000000000000010.checkpoint.parquet
│ └── _last_checkpoint
├── part-00000-...parquet
└── part-00001-...parquet

Each JSON log file records:
• add: New Parquet files added (with stats: min/max per column, null count)
• remove: Files logically deleted (marked as removed, not physically deleted)
• metaData: Schema, partition columns, configuration
• protocol: Reader/writer version requirements
• commitInfo: Operation type, timestamp, user, notebook ID

Key insight: Files are NEVER modified — only new files are added and old files are marked "removed" in the log. Physical deletion happens only during VACUUM.
Q 04
How does time travel work in Delta Lake when you query a historical version?
Answer: B

Time travel is elegantly simple because of the append-only log:

1. You request version N: spark.read.format("delta").option("versionAsOf", 5).load("/path")
2. Delta reads log entries 0 through N
3. Builds a "snapshot" — the set of active Parquet files at version N (files added up to N, minus files removed up to N)
4. Reads exactly those Parquet files — no data reconstruction needed

Supported syntax:
# By version number
df = spark.read.format("delta").option("versionAsOf", 10).load("/delta/events")

# By timestamp
df = spark.read.format("delta").option("timestampAsOf","2024-06-01").load("/delta/events")

# SQL
SELECT * FROM events VERSION AS OF 10
SELECT * FROM events TIMESTAMP AS OF '2024-06-01'

Limitation: Time travel only works as long as old Parquet files haven't been deleted by VACUUM (default retention: 7 days). Attempting to time-travel past the VACUUM horizon raises an error.
Q 05
How often is a Parquet checkpoint file created in the Delta transaction log?
Answer: B

Without checkpoints, reading a Delta table with 10,000 commits would require reading 10,000 JSON log files sequentially — extremely slow.

Checkpoints solve this: Every 10 commits (default), Delta writes a compact Parquet file that represents the COMPLETE state of the table at that version. The _last_checkpoint file records the latest checkpoint version.

On table open, Delta:
1. Reads _last_checkpoint → finds latest checkpoint (e.g., version 990)
2. Loads the checkpoint Parquet (represents full state at v990)
3. Reads only JSON files AFTER v990 (v991...v999)
→ Avoids reading all 1000 log files

Change the interval:
ALTER TABLE my_table SET TBLPROPERTIES ('delta.checkpointInterval' = '50')

Checkpoint contains: All active "add" entries (the complete set of live Parquet files + their column statistics), protocol, and metadata — the same info as replaying all JSON logs from scratch, but in a single efficient read.
Q 06
How do you read a Delta table's Parquet data files directly (bypassing the transaction log)?
Answer: B

Delta data files are standard Parquet — any Parquet reader can open them. However, reading them directly with spark.read.parquet("/path") is dangerous in production:

Problems with bypassing the log:
• You may read files that Delta has marked as "removed" (from deletes/updates) — seeing ghost data
• You may miss files still being written — seeing partial inserts
• No column statistics for data skipping — slower queries
• No schema enforcement — type mismatches silently accepted

Always use Delta format in production:
# Correct: respects transaction log
spark.read.format("delta").load("/delta/events")
spark.table("my_delta_table")

# Also valid for checkpoint inspection:
spark.read.parquet("/delta/events/_delta_log/00000000000000000010.checkpoint.parquet")

Use case for direct Parquet read: Debugging — reading the checkpoint Parquet to inspect active file list and column statistics without going through Spark SQL.
Log Internals
Q 07
What is the format and content structure of a Delta Lake transaction log JSON file?
Answer: B

Each .json log file is NDJSON — one JSON object per line, each representing one atomic action:

{"commitInfo":{"timestamp":1704067200000,"operation":"WRITE","operationParameters":{"mode":"Append"},"isolationLevel":"Serializable","isBlindAppend":true}}
{"protocol":{"minReaderVersion":1,"minWriterVersion":2}}
{"metaData":{"id":"abc123","format":{"provider":"parquet"},"schemaString":"{...}","partitionColumns":["date"]}}
{"add":{"path":"date=2024-01-01/part-00000-xyz.parquet","size":1048576,"stats":"{\"numRecords\":50000,\"minValues\":{\"amount\":0.01},\"maxValues\":{\"amount\":9999.99}}","dataChange":true}}
{"remove":{"path":"date=2024-01-01/part-old-abc.parquet","deletionTimestamp":1704067200000,"dataChange":true}}

Action types:
• commitInfo: Who, what operation, when, parameters
• add: New file with path, size, partition values, column stats
• remove: File being logically deleted (still physically present until VACUUM)
• metaData: Schema, partition columns, table properties
• protocol: Minimum reader/writer versions required
• cdc: Change data capture files (when CDC enabled)
Q 08
What is the format of a Delta Lake checkpoint file and why is it different from the JSON log files?
Answer: B

Checkpoints are stored as Parquet files: _delta_log/00000000000000000010.checkpoint.parquet

Why Parquet instead of JSON?
• Parquet is columnar — reading only the "path" column of 1M file entries is fast
• Parquet has built-in compression — much smaller than equivalent JSON
• Parquet supports predicate pushdown — can filter by partition value

What's IN the checkpoint:
The checkpoint contains only the NET state — only "add" entries for files that are STILL active (removes have been applied). It does NOT contain commitInfo or intermediate removes. Think of it as: "if you started fresh, these are the Parquet files that make up the table."

Multi-part checkpoints (Delta 2.0+): For very large tables with millions of files, the checkpoint can be split into multiple Parquet files:
00000000000000001000.checkpoint.0000000001.0000000003.parquet
00000000000000001000.checkpoint.0000000002.0000000003.parquet
00000000000000001000.checkpoint.0000000003.0000000003.parquet

Tip: You can read checkpoint files directly with spark.read.parquet("_delta_log/...checkpoint.parquet") to debug what files are in the table's active set.
Operations & DML
Q 09
Which of the following is NOT a supported operation in Delta Lake?
Answer: C

Delta Lake is not a relational database — it has no concept of foreign keys or CASCADE constraints. TRUNCATE WITH CASCADE is a PostgreSQL/SQL Server concept.

Delta Lake's actual operations:
• CREATE TABLE: CREATE TABLE t USING DELTA or df.write.format("delta").save(path)
• INSERT / APPEND: df.write.mode("append").format("delta").save(path)
• OVERWRITE: df.write.mode("overwrite").format("delta").save(path)
• UPDATE: UPDATE t SET col=val WHERE condition
• DELETE: DELETE FROM t WHERE condition
• MERGE: Upsert, conditional insert/update/delete
• OPTIMIZE: Compact small files, Z-Order
• VACUUM: Remove old unreferenced files
• RESTORE: Roll back to a specific version
• CLONE: Shallow (metadata only) or deep (copy data) clone
• DESCRIBE HISTORY: Audit log of all operations
Q 10
When you run UPDATE events SET status='processed' WHERE date='2024-01-01' in Delta Lake, what physically happens?
Answer: B

Parquet files are immutable — you cannot update individual bytes. Delta handles this with a copy-on-write (CoW) approach by default:

UPDATE execution steps:
1. Scan: Find which Parquet files contain rows matching date='2024-01-01' (using column stats to skip irrelevant files)
2. Read + Rewrite: For each affected file, read all rows, apply the update, write a new Parquet file
3. Log commit: Atomically write a log entry with:
  • remove actions for old files
  • add actions for new files
4. Old files still exist on disk until VACUUM runs

This means: A small update to one row in a 10GB partition file causes the ENTIRE partition file to be rewritten. This is why:
• Small, frequent updates are expensive in Delta (consider Deletion Vectors for this!)
• Batch updates are far more efficient than row-by-row updates
• Merge-on-Read (MoR) with Deletion Vectors reduces rewrite cost for deletes/updates
Architecture
Q 11
In a traditional Data Lake solution architecture, what is the primary challenge that leads to the "data swamp" problem?
Answer: B

Traditional Data Lake architecture (S3/ADLS + Spark/Hive) faces these core reliability problems:

1. No atomicity: A Spark job writing 1000 files fails at file 700 → lake has partial data. No rollback. Next reader sees corrupt state.

2. No schema enforcement: Anyone can write any schema. Data engineer writes amount as string; analyst expects decimal → silent errors downstream.

3. No isolation: Reader running a 2-hour query sees data being modified by a concurrent writer → dirty reads.

4. Undiscoverable data: Without a catalog, files are just blobs. Which files are current? Which are stale? Unknown.

5. No update/delete: To correct a row, you must rewrite entire partitions manually.

Traditional Data Lake stack:
• Storage: S3/ADLS/GCS
• Ingestion: Kafka → Spark Streaming
• Processing: Spark/Hive
• Query: Presto/Athena/Hive
• Catalog: Glue/Hive Metastore

Delta Lake addresses ALL of the above.
Q 12
In a Delta Lake solution architecture, what replaces the traditional "job failure = corrupt state" problem?
Answer: B

Delta Lake write protocol (simplified):

1. Writer reads latest table version (e.g., v5)
2. Writer computes which files to add/remove
3. Writer uploads new Parquet data files to storage
4. Writer attempts to atomically create 00000000000000000006.json (v6 commit)
5a. Success → commit exists → table is now v6 → new files are "visible"
5b. Failure → log file never written → old files remain active → orphan new files exist but are invisible (cleaned up by VACUUM)

Conflict detection (Optimistic Concurrency):
If two writers both try to create v6:
• Object stores (S3, ADLS) provide atomic put-if-absent for small files
• First writer wins, creates v6.json
• Second writer's put fails → checks if conflict is resolvable → if the two writers touched different files, Delta retries at v7; if same files, raises ConcurrentModificationException

This is why Delta supports concurrent reads with ZERO blocking and concurrent writes with conflict detection.
Q 13
In the Medallion Architecture, what is the typical data format in the Bronze layer?
Answer: B

The Medallion (Bronze/Silver/Gold) Architecture is a data design pattern for Delta Lake:

🥉 Bronze Layer (Raw):
• Source: Kafka, REST APIs, JDBC, files (CSV/JSON/XML/Avro)
• Format: Stored as Delta tables but containing raw data — minimal or no transformation
• Added metadata: ingestion timestamp, source file name, batch ID
• Schema: Often permissive (all strings) to avoid ingestion failures
• Purpose: Immutable landing zone — "source of truth" for replay
• Retention: Long-term (often years)

Best practice for Bronze:
raw_df\
  .withColumn("_ingest_timestamp", current_timestamp())
  .withColumn("_source_file", input_file_name())
  .write.format("delta")\
  .mode("append")\
  .save("/lake/bronze/events")

Key principle: Never delete from Bronze. If upstream data changes, you can always re-derive Silver/Gold from Bronze. Bronze is your insurance policy.
Q 14
What characterizes data in the Silver layer of the Medallion Architecture?
Answer: B

🥈 Silver Layer (Cleansed / Conformed):
• Input: Bronze Delta tables
• Format: Delta tables with enforced schema
• Transformations applied:
  – Type casting (string → int, string → date)
  – Deduplication (remove exact or fuzzy duplicates)
  – Null handling (drop, fill, flag)
  – Data validation (reject out-of-range values)
  – Business key resolution (normalize customer IDs)
  – Joins (enrich events with dimension data)
• Schema: Strictly enforced, documented
• Users: Data scientists, ML engineers, advanced analysts
• SLA: Updated within minutes to hours of Bronze

Silver loading pattern:
bronze_df = spark.read.format("delta").load("/lake/bronze/orders")
silver_df = bronze_df\
  .dropDuplicates(["order_id"])\
  .withColumn("amount", col("amount_str").cast("decimal(18,2)"))\
  .filter(col("order_date").isNotNull())
silver_df.write.format("delta").mode("overwrite").save("/lake/silver/orders")
Q 15
What type of data and transformations belong in the Gold layer of the Medallion Architecture?
Answer: B

🥇 Gold Layer (Business-ready / Serving):
• Input: Silver Delta tables
• Format: Delta tables, heavily optimized (Z-Ordered, compacted)
• Transformations: GROUP BY, window functions, complex aggregations, business logic
• Models: Star schema fact/dimension tables, wide denormalized tables, cubes
• Users: BI tools (Power BI, Tableau), executives, dashboards, APIs
• Update frequency: Hourly, daily (depending on business need)

Example Gold table:
# Daily sales summary (Gold)
silver_orders\
  .groupBy("region", "product_category", "order_date")\
  .agg(
    sum("amount").alias("total_revenue"),
    count("order_id").alias("order_count"),
    avg("amount").alias("avg_order_value")
  )\
  .write.format("delta")\
  .mode("overwrite")\
  .partitionBy("order_date")\
  .save("/lake/gold/daily_sales_summary")

Separation of concerns: Errors at any layer don't propagate upward retroactively. Bronze mistakes can be fixed, Silver re-derived, Gold rebuilt — without touching source systems.

Q 16
Which method correctly creates an external Delta table with partitioning on a cloud storage path?
Answer: B

Multiple ways to create a Delta table:

1. DataFrame write (creates data + log):
df.write.format("delta")\
  .mode("overwrite")\
  .partitionBy("year", "month")\
  .save("s3://my-bucket/delta/events")

2. SQL CREATE TABLE (schema only, no data):
CREATE TABLE analytics.events (
  event_id STRING,
  user_id BIGINT,
  amount DECIMAL(18,2),
  event_date DATE
) USING DELTA
PARTITIONED BY (event_date)
LOCATION 's3://my-bucket/delta/events'

3. DeltaTableBuilder API:
DeltaTable.createOrReplace(spark)\
  .tableName("analytics.events")\
  .addColumn("event_id", "STRING")\
  .addColumn("amount", "DECIMAL(18,2)")\
  .partitionedBy("event_date")\
  .location("s3://bucket/events")\
  .execute()

Managed vs External: Without LOCATION → managed table (Spark owns data). With LOCATION → external table (DROP TABLE keeps data on disk).
Q 17
What is a DeltaTable instance in PySpark, and when would you use it instead of SQL?
Answer: B

from delta.tables import DeltaTable

# Load by path
dt = DeltaTable.forPath(spark, "/delta/events")

# Load by table name
dt = DeltaTable.forName(spark, "analytics.events")

DeltaTable-specific operations:

# MERGE (upsert) — most common use
dt.alias("target").merge(
  source_df.alias("source"),
  "target.id = source.id"
).whenMatchedUpdateAll()\
  .whenNotMatchedInsertAll()\
  .execute()

# View history
dt.history().show(truncate=False)

# VACUUM
dt.vacuum(retentionHours=168) # 7 days

# RESTORE
dt.restoreToVersion(5)
dt.restoreToTimestamp("2024-01-01 00:00:00")

# Get underlying DataFrame
df = dt.toDF()

Use DeltaTable when you need MERGE logic, programmatic history access, or operations that benefit from Python control flow (conditional merges, dynamic source construction).

ACID & Reliability
Q 19
How does Delta Lake achieve Isolation (the "I" in ACID) for concurrent readers and writers?
Answer: B

Delta Lake implements Snapshot Isolation (a form of MVCC — Multi-Version Concurrency Control):

Reader behavior:
1. Reader starts a query → reads latest committed version (e.g., v10)
2. Builds snapshot: set of Parquet files active at v10
3. During the query, writer commits v11 (new files added)
4. Reader continues seeing v10 — completely unaffected by v11
5. Next reader query will see v11

Writer conflict detection:
• Two writers start at v10
• Writer A commits → creates v11.json
• Writer B tries to commit → detects v11 exists
• Delta checks: do v11's changes conflict with Writer B's changes?
  – If no overlap → Writer B retries and commits v12 (success)
  – If overlap (same rows modified) → ConcurrentModificationException

Isolation levels supported:
• Serializable (default for writes): Strictest — no write-write conflicts
• WriteSerializable: Allows some concurrent writes that don't logically conflict
• Reads: Always Snapshot Isolation — ZERO reader-writer conflicts
Q 20
What happens when you try to write a DataFrame with an extra column to a Delta table with schema enforcement enabled?
Answer: B

Schema enforcement (also called schema validation) is Delta Lake's default protection against accidental schema changes:

What triggers an AnalysisException:
• DataFrame has a column NOT in the table schema
• DataFrame has different data type for an existing column (e.g., long vs string)
• DataFrame is missing a non-nullable column

Example:
# Table schema: id INT, name STRING
bad_df = spark.createDataFrame([(1, "Alice", "New York")], ["id","name","city"])
bad_df.write.format("delta").mode("append").save("/delta/users")
# → AnalysisException: A schema mismatch detected when writing to the Delta table
# → Extra column: city

How to intentionally add the column (schema evolution):
bad_df.write.format("delta")\
  .mode("append")\
  .option("mergeSchema", "true")\
  .save("/delta/users")
# Now table has: id, name, city (city is null for old rows)

Schema enforcement is a feature, not a bug — it prevents data quality issues from silently propagating into your lake.
Q 21
Which type of schema change CANNOT be handled automatically by Delta Lake's schema evolution, even with mergeSchema=true?
Answer: C

Delta Lake supports additive schema evolution automatically via mergeSchema=true or autoMerge=true:

✅ Supported automatically:
• Adding new columns (nullable)
• Widening numeric types: INT → LONG, FLOAT → DOUBLE
• Adding new fields to existing StructType columns
• Changing nullable to non-nullable (with caution)

❌ NOT supported (requires manual intervention):
• Incompatible type changes: STRING → INT, LONG → INT (data loss risk)
• Removing columns (Delta doesn't automatically drop columns)
• Renaming columns (treated as drop + add, not a rename)
• Changing partition columns

For unsupported changes — use overwrite:
new_df.write.format("delta")\
  .mode("overwrite")\
  .option("overwriteSchema", "true")\
  .save("/delta/table")
# Overwrites both data AND schema — use carefully!

Production approach: Treat schema changes as migrations — version them, test them in lower environments, and document them in the table's DESCRIBE HISTORY.
Query Optimization
Q 22
How does Z-Ordering improve query performance, and what type of queries benefit most?
Answer: B

How Z-Ordering works:
Z-Ordering (Z-curve / Morton curve) is a space-filling curve that maps multi-dimensional data to one dimension while preserving locality. When applied to columns A and B, rows with similar (A, B) values cluster together in files.

OPTIMIZE events ZORDER BY (user_id, event_date)

Data skipping mechanism:
Each Parquet file in Delta stores column statistics (min/max per column). When you query WHERE user_id = 12345:
1. Delta reads min/max user_id for each file
2. Skips files where 12345 is outside [min, max]
3. Only reads files that could contain user_id = 12345

Without Z-Order: user_ids scattered across all files → data skipping ineffective → scan all files
With Z-Order: user_ids clustered → most files skipped → only read 1-2% of data

Best columns for Z-Ordering:
• High-cardinality columns in WHERE clauses: user_id, product_id, transaction_id
• Time columns with range queries: event_timestamp
• Limit to 1-4 columns (effectiveness decreases with more columns)
Q 23
What is the difference between OPTIMIZE and VACUUM in Delta Lake?
Answer: B

OPTIMIZE — improves READ performance:
OPTIMIZE my_table -- compact to 1GB files
OPTIMIZE my_table ZORDER BY (user_id) -- compact + co-locate
OPTIMIZE my_table WHERE date='2024-01-01' -- partition-targeted
• Reads many small files, writes fewer large files (~1GB target)
• Old small files get "remove" entries in the log (still exist on disk)
• Small files problem: 1000 × 1MB files vs 1 × 1GB file → 1000x more file open overhead
• Does NOT delete anything from disk

VACUUM — reclaims STORAGE space:
VACUUM my_table RETAIN 168 HOURS -- delete files older than 7 days
VACUUM my_table RETAIN 0 HOURS -- DANGEROUS: kills time travel entirely
• Physically deletes Parquet files marked as "removed" in the log
• Also deletes orphan files (files not referenced by any log entry)
• Default retention: 7 days (168 hours)
• After VACUUM, time travel before the retention window is impossible

Order matters: Run OPTIMIZE regularly (hourly/daily). Run VACUUM periodically (weekly) after verifying no long-running queries need old files.
Q 24
Two Spark jobs simultaneously try to DELETE different rows from the same Delta table. What happens?
Answer: B

Delta Lake uses optimistic concurrency control — it assumes conflicts are rare and checks at commit time:

Scenario: Job A deletes rows from partition P1; Job B deletes rows from partition P2
1. Job A reads v10, removes files in P1, tries to commit v11 → SUCCESS
2. Job B reads v10, removes files in P2, tries to commit v11 → v11 exists!
3. Delta checks: did Job B's read-set overlap with Job A's write-set?
  → P1 ≠ P2: NO overlap → Job B retries and commits as v12 ✅

Scenario: Both jobs delete from the SAME partition file
1. Job A commits v11 (removing file X)
2. Job B tries to commit — it also read file X and wants to remove it
3. Delta detects: Job A already removed file X → conflict!
→ Job B raises ConcurrentDeleteReadException
→ Must be retried (re-read latest version, re-execute delete)

Exception types:
• ConcurrentDeleteReadException: Concurrent delete on same files
• ConcurrentDeleteDeleteException: Both try to delete same file
• ConcurrentTransactionException: General conflict

Q 25
What is the most efficient pattern for incrementally loading new records into a Delta table without reprocessing historical data?
Answer: B

Pattern 1 — Delta as Streaming Source:
stream = spark.readStream.format("delta")\
  .load("/delta/bronze/events")
stream.writeStream.format("delta")\
  .option("checkpointLocation", "/checkpoints/silver_events")\
  .start("/delta/silver/events")
Delta's streaming source automatically tracks which versions have been processed via the checkpoint.

Pattern 2 — High-Water-Mark batch:
last_ts = spark.read.format("delta").load("/meta/watermarks")\
  .filter(col("pipeline")=="events").select("max_ts").collect()[0][0]

new_data = spark.read.format("delta").load("/delta/bronze/events")\
  .filter(col("updated_at") > last_ts)

# Upsert via MERGE
DeltaTable.forPath(spark, "/delta/silver/events")\
  .alias("t").merge(new_data.alias("s"), "t.id = s.id")\
  .whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()

Pattern 3 — Delta Change Data Feed (CDC):
changes = spark.read.format("delta")\
  .option("readChangeFeed","true")\
  .option("startingVersion", last_version)\
  .table("bronze_events")
# Each row has _change_type: insert/update_postimage/delete
Q 26
In a Delta MERGE operation, what happens if a source row matches multiple target rows?
Answer: B

Delta MERGE follows SQL standard semantics — a source row can match AT MOST one target row. If duplicates exist, you get: UnsupportedOperationException: Cannot perform Merge as multiple source rows matched and attempted to modify the same target row

Complete MERGE syntax:
DeltaTable.forPath(spark, "/delta/customers").alias("target")\
  .merge(
    updates_df.alias("source"),
    "target.customer_id = source.customer_id"
  )\
  .whenMatchedUpdate(set={"name":"source.name","email":"source.email"})\
  .whenNotMatchedInsertAll()\
  .whenNotMatchedBySourceDelete()\
  .execute()

Deduplicating source before MERGE:
deduped_source = updates_df\
  .withColumn("rn", row_number().over(
    Window.partitionBy("customer_id").orderBy(col("updated_at").desc())
  ))\
  .filter(col("rn") == 1).drop("rn")

MERGE clause types:
• whenMatchedUpdate / whenMatchedDelete
• whenNotMatchedInsert (target has no match)
• whenNotMatchedBySourceUpdate/Delete (source has no match — Delta 2.0+)
Q 27
How do you automate small file compaction in a production Delta Lake pipeline?
Answer: B

Strategy 1 — Databricks Auto Compaction (automatic):
ALTER TABLE my_table SET TBLPROPERTIES (
  'delta.autoOptimize.autoCompact' = 'true',
  'delta.autoOptimize.optimizeWrite' = 'true'
)
• Optimized Writes: Dynamically adjusts shuffle partitions to produce ~128MB files per write
• Auto Compaction: After each write, asynchronously compacts files that are too small

Strategy 2 — Scheduled OPTIMIZE job:
# Airflow / workflow task running nightly
spark.sql("OPTIMIZE analytics.events ZORDER BY (user_id)")
spark.sql("VACUUM analytics.events RETAIN 168 HOURS")

Strategy 3 — Partition-targeted OPTIMIZE after write:
date_val = datetime.now().strftime("%Y-%m-%d")
spark.sql(f"OPTIMIZE events WHERE date='{date_val}'")
# Only compacts today's partition, much faster

Target file size config: spark.databricks.delta.optimize.maxFileSize (default ~1GB). Adjust based on typical query patterns — smaller files for selective queries, larger for full scans.
Q 28
What is a Deletion Vector in Delta Lake, and how does it improve DELETE/UPDATE performance?
Answer: B

Deletion Vectors (DVs) were introduced in Delta Lake 2.3 and represent a fundamental shift from Copy-on-Write (CoW) to Merge-on-Read (MoR) for deletes and updates:

Without Deletion Vectors (Copy-on-Write):
DELETE 1 row from a 1GB Parquet file → entire 1GB file rewritten as new file → extremely expensive for frequent small deletes

With Deletion Vectors (Merge-on-Read):
DELETE 1 row from a 1GB Parquet file → write a tiny .bin bitmap file marking that row as deleted → original Parquet file unchanged → near-instant delete!

How reads work with DVs:
1. Reader loads Parquet file
2. Loads associated deletion vector bitmap
3. Filters out rows marked as deleted in the bitmap
→ Reader sees correct data without expensive rewrite

Enable Deletion Vectors:
ALTER TABLE my_table SET TBLPROPERTIES (
  'delta.enableDeletionVectors' = 'true'
)

Trade-offs:
• ✅ Writes: Much faster (no file rewrite)
• ⚠️ Reads: Slightly slower (must apply bitmap filter)
• Run OPTIMIZE periodically to physically remove deleted rows and eliminate DVs, restoring optimal read performance
