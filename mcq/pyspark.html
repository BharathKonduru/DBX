Core Fundamentals
Q 01
What is the primary difference between RDD, DataFrame, and Dataset in PySpark?
Answer: B

RDD (Resilient Distributed Dataset) is the foundational abstraction — a distributed collection of JVM objects with no schema, no optimization, and full control over execution. It's type-safe but verbose.

DataFrame adds a schema (column names + types) on top of RDD, enabling the Catalyst Optimizer to generate optimized query plans. However, it sacrifices compile-time type safety — errors surface at runtime, not compile time. In Python, everything is a DataFrame (there is no Python Dataset).

Dataset (Scala/Java only) combines the best of both: schema-aware like DataFrame + compile-time type safety like RDD. Under the hood, Dataset[Row] is literally a DataFrame.

Key insight: DataFrame/Dataset use the Tungsten binary format which is much more memory-efficient than Java object serialization in RDDs.
Q 02
How does PySpark achieve parallel processing across a cluster?
Answer: B

PySpark's parallelism model works as follows:

1. Driver runs your Python code and communicates with the JVM Spark engine via Py4J — a bridge that translates Python calls to JVM calls.

2. Data is split into partitions. Each partition is a logical chunk of data that can be processed independently.

3. The Spark Scheduler breaks jobs into stages and tasks. Each task processes one partition on one executor core.

4. Executors (JVM processes on worker nodes) run tasks in parallel. For Python UDFs, a separate Python worker process is spawned per executor, and data is serialized/deserialized via pickle (which is why Python UDFs are slower than native Spark functions).

The key takeaway: parallelism comes from data partitioning + distributed executors, NOT Python-level threading.
Q 03
Which real-world analogy BEST describes lazy evaluation in PySpark?
Answer: B

Lazy evaluation means transformations (filter, map, select) build a logical plan (DAG) but do NOT execute until an action (collect, count, write) is called.

The GPS analogy fits perfectly: You enter a destination (define transformations) and the GPS plans the full route (builds a DAG + optimizes it via Catalyst). The car only moves when you press Go (action is called). If traffic changes (data changes), it recalculates.

Benefits of lazy evaluation:
• Catalyst Optimizer sees the complete plan and can reorder, push down filters, eliminate redundant steps
• Avoids unnecessary computation — if you filter then limit(10), Spark won't process all records
• Pipeline fusion: multiple transformations can be combined into a single stage

SparkContext is the entry point to Spark functionality, representing the connection to the cluster. In Spark 2+, SparkSession wraps it. It's responsible for creating RDDs, broadcasting variables, accumulating metrics, and coordinating with the cluster manager.
Q 04
When processing a 500GB file in PySpark, which approach best handles memory constraints?
Answer: B

For large files, the correct strategy is:

1. Use columnar formats (Parquet, ORC) which support predicate pushdown — Spark reads only relevant columns/row groups, not the entire file.

2. Partition pruning: Store data partitioned by date/region so Spark skips irrelevant directories entirely.

3. Adjust partitions: Rule of thumb — each partition should be 128MB–200MB. For 500GB: ~2500–4000 partitions. Use spark.sql.files.maxPartitionBytes to control this.

4. Avoid collect() on large data — this brings everything to the driver and causes OOM errors.

5. Use streaming reads for incremental processing with spark.readStream.

6. Set spark.sql.adaptive.enabled=true (AQE) to dynamically adjust partition sizes based on actual data.
Q 05
Which of these is a TRANSFORMATION, not an action, in PySpark?
Answer: C

Transformations are lazy — they return a new DataFrame/RDD and add a step to the logical plan without executing anything:
• select(), filter(), groupBy(), join(), withColumn(), orderBy(), agg()

Actions trigger execution of the entire DAG:
• count() — returns a number
• collect() — returns all rows to driver
• write — saves data to storage
• show(), take(n), first(), foreach()

The tricky part: groupBy().agg() looks like it's computing something, but it's still building a logical plan. Only when you call show() or collect() on the result does actual computation happen. This is the core of lazy evaluation.
Q 06
How does Spark determine the number of partitions when reading a file from HDFS?
Answer: C

Spark's partitioning logic:

For HDFS/S3 files: Default partition size = HDFS block size (128MB). A 1GB file → ~8 partitions. Controlled by spark.sql.files.maxPartitionBytes (default 128MB).

For RDDs: sc.textFile(path, minPartitions) — you can hint at minimum partitions but Spark may use more based on block size.

For JDBC sources: You control partitions via numPartitions, partitionColumn, lowerBound, upperBound.

Key configs:
• spark.sql.files.maxPartitionBytes — max bytes per partition
• spark.sql.files.openCostInBytes — cost to open a file (small files get merged)
• spark.default.parallelism — for RDD operations, defaults to total executor cores

With AQE enabled, Spark can coalesce shuffle partitions dynamically after seeing actual data sizes.
Q 07
What makes PySpark fault-tolerant when a node fails mid-computation?
Answer: C

Spark's fault tolerance is based on lineage graphs (DAGs), not data replication:

Every RDD knows exactly how it was created — its parent RDDs and the transformation applied. This chain is the lineage. If a partition is lost (executor dies), Spark traces back through the lineage to recompute only the lost partition from its source data.

Wide vs Narrow dependencies matter here:
• Narrow dependency (e.g., map, filter): Each child partition depends on one parent partition. Recovery is cheap — recompute just that partition.
• Wide dependency (e.g., groupBy, join): Child partitions depend on ALL parent partitions. Recovery may require recomputing many partitions or previous shuffle stages.

For long lineage chains (streaming, iterative algorithms), use checkpointing (df.checkpoint()) to truncate lineage and save state to reliable storage, preventing expensive full recomputation.
Q 08
When should you use a broadcast variable instead of a regular variable in Spark?
Answer: B

Without broadcast: Spark serializes the variable and sends it with EVERY task. If you have 10,000 tasks and a 10MB lookup dict, that's 100GB of network transfer.

With broadcast:
lookup = spark.sparkContext.broadcast({"US": "United States", ...})
Spark sends the data once per executor (using an efficient P2P BitTorrent-like protocol), where it's cached in memory. All tasks on that executor share the same copy.

Best use cases:
• Lookup/dimension tables for joins (this is a "broadcast join")
• Configuration dictionaries
• ML model weights for distributed inference

Size limit: Controlled by spark.broadcast.blockSize and spark.sql.autoBroadcastJoinThreshold (default 10MB for auto-broadcast joins). For larger tables, manually broadcast and use hints: df.join(broadcast(small_df), ...).

Access via lookup.value inside UDFs.
Q 09
What is the key behavioral difference between accumulators and broadcast variables?
Answer: B

Broadcast variables: Driver → Executors (read-only). Used to push large, static data to workers.

Accumulators: Executors → Driver (aggregation only). Workers ADD to the accumulator; only the driver can READ the final value.

error_count = sc.accumulator(0)
df.foreach(lambda row: error_count.add(1) if row["status"] == "ERROR" else None)
print(error_count.value) # Only valid on driver

Critical gotcha: Accumulators in transformations (lazy operations) may be updated multiple times due to task retries or re-execution. This leads to double-counting. Accumulators in actions are more reliable. Always treat accumulator values as approximate metrics, not exact counts.

Custom accumulators: Extend AccumulatorParam to create custom aggregation types (e.g., sets, dictionaries).
Q 10
Which filter approach is most performant when working with PySpark DataFrames?
Answer: C

Native Column expressions (col(), SQL strings) are processed entirely within the JVM by the Catalyst Optimizer and Tungsten Execution Engine — no Python involved during actual execution.

Why Python lambdas/UDFs are slower:
1. Data must be serialized from JVM → Python (pickle)
2. Python processes the data row-by-row
3. Results serialized back Python → JVM
This serialization overhead can make Python UDFs 3–100x slower than equivalent native expressions.

Performance ranking (fastest to slowest):
1. Built-in Spark functions (col, lit, functions.*)
2. Spark SQL string expressions ("age > 30")
3. Pandas UDFs (vectorized, uses Apache Arrow)
4. Python UDFs (row-by-row, highest overhead)

Always prefer built-in functions from pyspark.sql.functions over UDFs when possible.
DataFrame Operations
Q 11
A job finishes with 1000 shuffle partitions but most are tiny (few KB). What's the best fix?
Answer: B

coalesce(n): Reduces partitions by merging existing ones WITHOUT a full shuffle. It combines adjacent partitions on the same executor, making it extremely efficient for downsizing. Limitation: cannot increase partition count.

repartition(n): Performs a FULL shuffle — redistributes ALL data evenly across n partitions. Use when you need:
• More partitions (scale up)
• Even distribution (fix data skew)
• Partitioning by a specific column: repartition(n, col("country"))

Decision rule:
• Reducing partitions → coalesce() (no shuffle, fast)
• Increasing/rebalancing → repartition() (full shuffle, expensive)

AQE solution: With Spark 3+, enable spark.sql.adaptive.enabled=true and spark.sql.adaptive.coalescePartitions.enabled=true — Spark automatically coalesces shuffle partitions based on target size (spark.sql.adaptive.targetPostShuffleInputSize, default 64MB).
Q 12
Which method correctly fills null numeric values with the column mean in PySpark?
Answer: C

PySpark doesn't have a single df.mean() method. You must compute means first:

from pyspark.sql import functions as F
means = df.select([F.mean(c).alias(c) for c in numeric_cols]).collect()[0]
fill_dict = {c: means[c] for c in numeric_cols if means[c] is not None}
df_filled = df.na.fill(fill_dict)

Key null-handling functions:
• df.na.drop(how="any"/"all") — drop rows with nulls
• df.na.fill(value) or df.na.fill({"col": val}) — fill nulls
• df.na.replace(old, new) — replace specific values
• F.coalesce(col1, col2) — return first non-null
• F.when(col.isNull(), default).otherwise(col) — conditional fill

MLlib Imputer: For sophisticated imputation (mean/median/mode), use pyspark.ml.feature.Imputer which handles multiple columns at once as a pipeline stage.
Q 13
What does df.withColumn("new_col", col("salary") * 1.1) actually do internally?
Answer: B

DataFrames are immutable. withColumn() returns a NEW DataFrame with the transformation added to the logical plan. The original df is completely unchanged.

df2 = df.withColumn("salary_bumped", col("salary") * 1.1) # df still has no "salary_bumped" column # df2 has the new column in its plan (not yet computed)

Performance warning: Chaining many withColumn() calls can create deeply nested logical plans. For adding many columns at once, prefer:
df.select("*", (col("a") + 1).alias("b"), (col("c") * 2).alias("d"))

Available column operations:
• withColumn(name, expr) — add/replace column
• withColumnRenamed(old, new) — rename
• drop(col_name) — remove column
• withMetadata(name, metadata) — attach metadata
Q 14
In a left join between df_orders and df_customers, what happens to orders with no matching customer?
Answer: B

df_orders.join(df_customers, on="customer_id", how="left")

In a LEFT JOIN, ALL rows from the left DataFrame (df_orders) are preserved. For unmatched rows, the right DataFrame's columns are filled with null.

All join types:
• "inner" — only matching rows
• "left" or "left_outer" — all left rows, nulls for unmatched right
• "right" or "right_outer" — all right rows, nulls for unmatched left
• "full" or "full_outer" — all rows from both, nulls where unmatched
• "left_semi" — left rows that HAVE a match (no right columns in output)
• "left_anti" — left rows that DON'T have a match (like NOT EXISTS)
• "cross" — cartesian product

Tip: left_semi and left_anti are very powerful for filtering without including the right table's columns — great for deduplication checks.
Q 15
What is the key difference between a temp view and a global temp view in PySpark?
Answer: B

Temporary View:
df.createOrReplaceTempView("sales")
• Scoped to the current SparkSession
• Dropped when the SparkSession ends
• Referenced as: spark.sql("SELECT * FROM sales")

Global Temporary View:
df.createOrReplaceGlobalTempView("sales_global")
• Scoped to the entire Spark application (survives SparkSession restarts)
• Stored in the special global_temp database
• Referenced as: spark.sql("SELECT * FROM global_temp.sales_global")
• Dropped when the Spark application terminates

Neither is truly persistent — both exist only in memory for the application lifetime. For persistence across application runs, use df.write.saveAsTable("table_name") in the Hive metastore.
Q 16
Which window function correctly calculates a running total of sales per region, ordered by date?
Answer: B

from pyspark.sql import Window
from pyspark.sql import functions as F

w = Window.partitionBy("region").orderBy("date").rowsBetween(Window.unboundedPreceding, Window.currentRow)
df = df.withColumn("running_total", F.sum("sales").over(w))

Key window frame types:
• rowsBetween(start, end) — physical rows
• rangeBetween(start, end) — value-based range
• Window.unboundedPreceding — start of partition
• Window.unboundedFollowing — end of partition
• Window.currentRow — current row

Common window functions:
• Ranking: rank(), dense_rank(), row_number(), ntile(n)
• Analytics: lag(col, n), lead(col, n), first(), last()
• Aggregates: sum(), avg(), min(), max(), count() — all work with .over(window)
Q 17
What is the fastest way to register and use a UDF in PySpark for DataFrame operations?
Answer: B

Standard Python UDF (slowest):
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType
@udf(returnType=StringType())
def upper_udf(s): return s.upper() if s else None
df.withColumn("upper_name", upper_udf(col("name")))

Pandas UDF (vectorized, ~5-100x faster):
from pyspark.sql.functions import pandas_udf
import pandas as pd
@pandas_udf("string")
def upper_pandas(s: pd.Series) -> pd.Series:
  return s.str.upper()
df.withColumn("upper_name", upper_pandas(col("name")))

Pandas UDFs use Apache Arrow for zero-copy data transfer between JVM and Python, processing entire batches/columns at once.

Register for SQL:
spark.udf.register("upper_sql", upper_udf)
spark.sql("SELECT upper_sql(name) FROM users")
Q 18
What is the practical difference between df.cache() and df.persist(StorageLevel.DISK_ONLY)?
Answer: C

cache() = persist(StorageLevel.MEMORY_AND_DISK) (note: for DataFrames in recent Spark versions it's MEMORY_AND_DISK; for RDDs it defaults to MEMORY_ONLY).

Storage levels:
• MEMORY_ONLY — RAM only, drop if insufficient (fast, risky)
• MEMORY_AND_DISK — RAM first, spill to disk (safe default)
• DISK_ONLY — disk only (slow but doesn't consume RAM)
• MEMORY_ONLY_SER — serialized in RAM (smaller footprint, more CPU)
• MEMORY_AND_DISK_SER — serialized, spills to disk
• OFF_HEAP — Tungsten off-heap memory

Both are lazy: materialization happens on first action after cache()/persist() is called. Unpersist with df.unpersist().

When to cache: DataFrames used 2+ times in the same job (iterative algorithms, multiple downstream aggregations). Don't cache if used only once — it wastes memory.
Q 19
Which format is BEST for large-scale analytical queries requiring column-level compression and predicate pushdown?
Answer: C

Parquet advantages:
• Columnar storage: Only reads queried columns (not entire rows). A query on 3 of 100 columns reads ~3% of data.
• Predicate pushdown: Row group statistics (min/max) skip irrelevant row groups without reading them.
• Efficient compression: Same-type column data compresses much better than mixed row data. Snappy/ZSTD compression gives 5-10x size reduction.
• Schema evolution: Supports adding/removing columns

Read/Write syntax:
# Write
df.write.partitionBy("year","month").parquet("/path")
# Read
spark.read.parquet("/path")
# CSV
spark.read.option("header","true").option("inferSchema","true").csv("/path")
# JSON
spark.read.json("/path")

Use CSV/JSON for: data exchange, human readability, smaller datasets. Use Parquet/ORC for: production data lakes, analytical workloads, storage efficiency.
Q 20
How do you run a SQL query on a PySpark DataFrame?
Answer: B

You cannot directly query a DataFrame by variable name in SQL — you must register it as a named view first:

df.createOrReplaceTempView("employees")
result = spark.sql("""
  SELECT dept, AVG(salary) as avg_sal
  FROM employees
  WHERE hire_date > '2020-01-01'
  GROUP BY dept
  HAVING avg_sal > 75000
""")

The result is itself a DataFrame — fully interoperable with the DataFrame API. You can chain DataFrame operations after SQL queries or convert SQL results to temp views for further SQL queries.

Spark SQL supports: Standard SQL + Spark-specific functions, CTEs, subqueries, window functions, LATERAL VIEW, PIVOT, UNPIVOT, and many analytical functions. It's executed through the same Catalyst Optimizer as the DataFrame API.
Spark SQL & Optimization
Q 21
What does the Catalyst Optimizer do that makes Spark SQL significantly faster than raw RDD code?
Answer: B

Catalyst is a query optimization framework with 4 phases:

1. Analysis: Resolves column names, types, and table references. Produces an analyzed logical plan.

2. Logical Optimization: Applies rule-based transformations:
• Predicate pushdown: Move filters as close to data source as possible
• Projection pruning: Read only needed columns
• Constant folding: Evaluate 1 + 2 at plan time → 3
• Null propagation, Boolean simplification, subquery elimination

3. Physical Planning: Generates multiple physical plans, estimates cost using statistics, picks the best (e.g., broadcast join vs sort-merge join).

4. Code Generation: Tungsten generates optimized JVM bytecode for the physical plan — effectively compiling your query to machine-optimal code.

This is why the same DataFrame operation can be 10-100x faster than equivalent RDD code: RDDs bypass ALL of this optimization.
Q 22
What is the risk of using inferSchema=True when reading CSV files in production?
Answer: B

Issues with inferSchema:
1. Extra scan: Spark must read all data once to infer types, doubling I/O cost
2. Type mismatches: "001" → inferred as LongType (loses leading zeros); sparse columns with few non-null values → may infer wrong type
3. Schema drift: If a new batch has a new column or changed type, inference gives a different schema → pipeline breaks

Production best practice — always define explicit schema:
from pyspark.sql.types import *
schema = StructType([
  StructField("id", StringType(), nullable=False),
  StructField("amount", DoubleType(), nullable=True),
  StructField("date", DateType(), nullable=True)
])
df = spark.read.schema(schema).csv("/path")

For Parquet/Avro/ORC: schema is embedded in the file — always preferred over CSV for production pipelines.
Q 23
When does Spark automatically choose a broadcast join over a sort-merge join?
Answer: B

Join strategy selection:

Broadcast Hash Join: Small table fits in memory → broadcast to ALL executors → each executor does a local hash lookup. No shuffle needed! Best performance.
• Threshold: spark.sql.autoBroadcastJoinThreshold=10MB (increase for larger tables)
• Force it: df_large.join(broadcast(df_small), "key")

Sort-Merge Join (default for large tables): Both sides are shuffled and sorted by join key, then merged. Expensive but scalable.

Shuffle Hash Join: One side is hashed into memory partitions. Better than sort-merge when one side is moderately small.

Cartesian Join: Cross join — avoid unless intentional.

With AQE: Spark 3+ can switch join strategies at runtime based on actual shuffle sizes, converting sort-merge to broadcast join dynamically if the runtime size fits.
Q 24
How do you create a persistent table in Spark SQL that survives application restarts?
Answer: B

df.write.mode("overwrite").saveAsTable("analytics.sales_2024")

This creates a managed table: Spark manages both data and metadata. Data is stored in the warehouse directory (HDFS or cloud storage), and metadata is registered in the Hive Metastore.

Or with SQL:
spark.sql("CREATE TABLE analytics.sales_2024 USING PARQUET AS SELECT * FROM temp_sales")

External vs Managed tables:
• Managed: saveAsTable() — Spark owns data. DROP TABLE deletes data.
• External: LOCATION '/path' — Spark only owns metadata. DROP TABLE keeps data.

spark.sql("""CREATE EXTERNAL TABLE ext_sales
USING PARQUET LOCATION 's3://bucket/sales/'""")

Once saved, query without re-registering: spark.table("analytics.sales_2024")
Q 25
How does dynamic partition pruning (DPP) improve query performance in Spark 3+?
Answer: B

Example scenario:
SELECT f.* FROM fact_sales f
JOIN dim_date d ON f.date_id = d.date_id
WHERE d.year = 2024

Without DPP: Spark scans ALL partitions of fact_sales then filters.
With DPP: Spark first executes the filter on dim_date to get matching date_ids (2024), then injects those values as a filter on fact_sales, pruning away all non-2024 partitions before they're read.

Requirements for DPP:
• spark.sql.optimizer.dynamicPartitionPruning.enabled=true (default in Spark 3+)
• The fact table must be partitioned by the join key
• The filtering table (dimension) must be small enough to broadcast

Performance impact: Can reduce I/O by 80-99% on star schema queries — one of the most impactful features in Spark 3. Works with Parquet, ORC, and Delta Lake.
Performance & Advanced Patterns
Q 26
A sort-merge join between two 50GB tables is slow. You can't increase cluster size. What's the best optimization?
Answer: B

Bucketing is the most powerful optimization for repeated joins on the same key:

# Write bucketed tables ONCE
df_orders.write.bucketBy(128, "customer_id").sortBy("customer_id")
  .saveAsTable("orders_bucketed")
df_customers.write.bucketBy(128, "customer_id").sortBy("customer_id")
  .saveAsTable("customers_bucketed")

# Future joins have NO shuffle
spark.table("orders_bucketed")
  .join(spark.table("customers_bucketed"), "customer_id")

When both tables have the same number of buckets and are bucketed on the join key, Spark skips the shuffle entirely — bucket N from table A joins with bucket N from table B locally. This eliminates the most expensive phase of sort-merge join.

Other optimizations for large joins:
• AQE: auto converts to broadcast if one side shrinks after filtering
• Salting: add random prefix to join key to fix data skew
• Predicate pushdown: filter before joining to reduce input sizes
Q 27
Your Spark job has 99 tasks completing in 2 min but 1 task taking 45 min. This is classic data skew. What's the best fix?
Answer: B

Salting technique for skewed groupBy:
# Add random salt to spread one key across N partitions
import random
from pyspark.sql import functions as F

salt_n = 10
df_salted = df.withColumn("salt", (F.rand() * salt_n).cast("int"))
  .withColumn("salted_key", F.concat(col("skewed_key"), F.lit("_"), col("salt")))

# First aggregation (parallel)
partial = df_salted.groupBy("salted_key").agg(F.sum("amount").alias("partial_sum"))

# Remove salt, final aggregation
result = partial.withColumn("key", F.split(col("salted_key"), "_")[0])
  .groupBy("key").agg(F.sum("partial_sum"))

AQE Skew Join Optimization (Spark 3+):
Enable spark.sql.adaptive.skewJoin.enabled=true — Spark automatically detects skewed partitions (larger than spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes) and splits them into smaller sub-partitions.
Q 28
How do you perform a multi-level aggregation (total sales per country AND per country+product) efficiently in a single SQL query?
Answer: B

spark.sql("""
SELECT country, product, SUM(sales)
FROM transactions
GROUP BY GROUPING SETS (
  (country), -- total per country
  (country, product) -- total per country+product
)
""")

ROLLUP: Generates hierarchical subtotals + grand total:
GROUP BY ROLLUP(country, product)
→ (country, product), (country), () [grand total]

CUBE: All possible combinations:
GROUP BY CUBE(country, product)
→ (country, product), (country), (product), ()

These are computed in a single data scan — far more efficient than running multiple queries. Use GROUPING(country) to distinguish subtotal rows (returns 1 for subtotals, 0 for regular rows). Essential for building summary tables and OLAP-style analytics.
Q 29
What does enabling spark.sql.cache.serializer and calling spark.catalog.cacheTable() do differently from df.cache()?
Answer: B

Lazy caching (df.cache()): Adds cache hint to logical plan. Data is NOT cached until first action.

Eager caching (spark.catalog.cacheTable("name")): Immediately triggers a scan and materializes the table into executor memory. No action needed.

SQL result cache:
spark.conf.set("spark.sql.cache.serializer", "org.apache.spark.sql.execution.columnar.InMemoryRelation")
# Or simpler:
spark.sql("CACHE TABLE sales_summary")

For repeated identical queries, Spark can return cached results directly. This is especially useful in BI tools where many users run the same dashboards repeatedly.

Uncaching:
spark.catalog.uncacheTable("sales_summary")
spark.sql("UNCACHE TABLE sales_summary")
df.unpersist()
ETL, Streaming & Production
Q 30
In a production ETL pipeline, what's the correct order of operations for reliability?
Answer: B

Production ETL architecture:

1. Extract: Read from source with explicit schema + error handling
2. Transform: Apply business logic, joins, aggregations
3. Validate: Check row counts, null rates, schema, value ranges. Fail fast with meaningful errors
4. Write idempotently:
  • Use Delta Lake for ACID transactions
  • Or write to temp path + atomic rename: _SUCCESS file pattern
  • Use mode("overwrite") with partition overwrite mode

Idempotency is critical for retry safety. A job that fails midway and is retried should produce the same result without duplicates.

Best practices:
• Medallion architecture: Bronze (raw) → Silver (cleaned) → Gold (aggregated)
• Log lineage metadata (source, row counts, checksums)
• Use circuit breakers for bad data
• Monitor data quality metrics over time
Q 31
In Structured Streaming, what is the default output mode for aggregation queries?
Answer: B

Three output modes in Structured Streaming:

Append: Only newly added rows are written. Rows in the result table are never changed. Works for non-aggregation queries or aggregations with watermark where results are finalized.

Complete: The entire result table is written every trigger. Required for aggregation without watermark (because any new data can change any aggregate). Expensive for large result sets.

Update: Only rows that changed since last trigger are output. Most efficient for aggregations — only outputs changed aggregates. Not supported by all sinks.

Mode compatibility:
| Query Type | Append | Complete | Update |
|---|---|---|---|
| No aggregation | ✅ | ❌ | ✅ |
| Agg with watermark | ✅ | ✅ | ✅ |
| Agg without watermark | ❌ | ✅ | ✅ |

Sinks: console, memory support all modes. Kafka, Delta Lake support append/update. File sink supports only append.
Q 32
What's the ideal partition size and count rule of thumb for optimal Spark performance?
Answer: B

Partition sizing guidelines:

• Target size: 128–256MB per partition in memory
• Count: 2–4x your total executor cores (ensures CPU stays busy even with stragglers)
• Too few partitions: Tasks too large, risk OOM, poor parallelism
• Too many partitions: Task scheduling overhead dominates, excessive shuffle files, driver bottleneck

Practical calculations:
• 1TB data, 128MB target → ~8192 partitions
• 100 cores → at least 200-400 partitions to keep all cores busy

Operation-specific tuning:
• Shuffle operations: Set spark.sql.shuffle.partitions (default 200, often needs tuning)
• Wide transformations increase partition count; coalesce() reduces
• With AQE enabled, Spark 3+ auto-coalesces small shuffle partitions

Rule: Start with 2 * num_cores, profile, then tune based on task time (ideal: 1–10 minutes per task).
Q 33
A Spark job is slow but you don't know why. What's the FIRST thing you should check?
Answer: B

Spark UI diagnostic checklist:

1. Jobs tab: Identify which job is slow → drill into stages
2. Stages tab: Look for:
  • Long stage duration
  • High shuffle read/write (indicates expensive operations)
  • Spill to disk (memory pressure)

3. Task timeline: Look for:
  • Straggler tasks (data skew)
  • High GC time (>10% = memory issue)
  • Scheduler delay (too many small tasks)
  • Fetch wait time (slow shuffle reads)

4. SQL tab: View physical plan, estimated vs actual row counts

Common findings and fixes:
• Skew → salting or AQE skew join
• GC pressure → increase memory or reduce object creation
• Many small tasks → coalesce, increase partition size
• Full stage scan → check predicate pushdown, add filters
• Lots of shuffles → cache intermediate results, bucket tables

A new column is added to a source CSV every week. How should your PySpark pipeline handle this schema evolution?

Q 35
What is the primary purpose of checkpointing in Spark Structured Streaming?
Answer: B

Checkpointing in Structured Streaming saves:

1. Source offsets: Exactly which records have been read (Kafka offsets, file positions) — enables picking up where we left off
2. Aggregation state: For stateful operations (counts, sessions), the current state of each group
3. Metadata: Committed batch IDs for end-to-end exactly-once semantics

query = (df.writeStream
  .format("delta")
  .option("checkpointLocation", "s3://bucket/checkpoints/query1")
  .outputMode("append")
  .start("/output"))

Without checkpointing: If the job fails and restarts, it either reprocesses all data or loses all accumulated state — neither is acceptable in production.

For RDDs (DStreams): dstream.checkpoint(Duration(30000)) truncates long lineage chains that would be expensive to recompute after failure.

Checkpoint location must be on fault-tolerant storage (HDFS, S3, ADLS) — not local filesystem.

You need to join a 500GB fact table with three 50GB dimension tables. What is the optimal strategy?
Answer: B

Multi-table join strategy:

1. Filter early: Apply all possible filters to each table BEFORE joining to minimize data size

2. Broadcast small dims: If filtered dimension is <10GB (or set threshold higher):

  .join(broadcast(dim3_filtered), "key3")

Q 38
What is the fundamental architectural difference between batch and stream processing in Spark?
Answer: B

Batch Processing:
• Bounded dataset (finite, known size)
• Runs periodically (hourly, daily)
• Can scan, sort, and aggregate entire dataset
• High latency (minutes to hours)
• High throughput — optimal for historical analysis

Stream Processing (Structured Streaming):
• Unbounded dataset (continuous, infinite)
• Runs continuously (milliseconds to seconds latency)
• Cannot scan entire history — must maintain state
• Spark implements as continuous micro-batches (default) or continuous processing mode

Structured Streaming mental model: Think of the incoming stream as rows appended to an infinite table. Each trigger, Spark processes new rows and updates the result table. This unifies batch and streaming — the same DataFrame API works for both!

df_stream = spark.readStream.format("kafka")...
result = df_stream.groupBy("key").count() # Same as batch!

Q 41
What makes Delta Lake's ACID transactions possible on a distributed file system like HDFS/S3?
Answer: B

Delta Lake transaction log architecture:

Every table has a _delta_log/ directory containing JSON files (one per transaction):
_delta_log/00000000000000000000.json ← initial table creation
_delta_log/00000000000000000001.json ← first write
_delta_log/00000000000000000010.checkpoint.parquet ← compacted log

Each JSON entry records: added files, removed files, schema, statistics.

ACID properties:
• Atomicity: Write new Parquet files, then atomically commit JSON entry. Failure = no commit = no change visible
• Consistency: Schema enforcement prevents corrupt writes
• Isolation: Optimistic concurrency — concurrent writes detect conflicts via version checks
• Durability: Committed entries on HDFS/S3 are durable

Benefits: Safe concurrent reads/writes, schema evolution, time travel, upserts with MERGE, CDC, audit history.

Q 43
How do you calculate a 7-day rolling average of daily sales per store using window functions?
Answer: B

from pyspark.sql import Window, functions as F

w = Window.partitionBy("store_id")
  .orderBy("date")
  .rowsBetween(-6, 0) # current row + 6 preceding rows = 7 rows

df = df.withColumn("rolling_avg_7d", F.avg("sales").over(w))

rowsBetween vs rangeBetween:
• rowsBetween(-6, 0): Exactly 7 physical rows (correct for daily data where each row = 1 day)
• rangeBetween(-6, 0): 7 units of the ORDER BY column's value — for dates ordered as integers, this means 6 days back. BUT if dates are stored as DateType, range-based windows can behave unexpectedly.

For sparse data (missing days): rowsBetween gives 7 data points regardless of gaps; rangeBetween gives a true calendar window but may include fewer rows.

Other rolling patterns:
• Last 30 days: rowsBetween(-29, 0)
• Trailing 3 rows: rowsBetween(-2, 0)
• Centered 5-row: rowsBetween(-2, 2)
Q 44
What are stateful operations in Structured Streaming, and why do they require special handling?
Answer: B

Stateful operations need to remember past data across micro-batches:
• Aggregations: Running count of users per session
• Stream-stream joins: Buffering one side waiting for the other
• Sessionization: Grouping events within a time window
• mapGroupsWithState / flatMapGroupsWithState: Custom stateful logic

The problem: Without bounds, state grows infinitely → OOM.

Solution — Watermarking:
df.withWatermark("event_time", "10 minutes")
  .groupBy(window(col("event_time"), "5 minutes"), col("user_id"))
  .count()
The watermark tells Spark: "Data more than 10 minutes late can be dropped." This bounds state — Spark evicts state older than max_event_time - watermark_delay.

State store: By default, state is stored in HDFS/checkpoint location (RocksDB state store available for better performance). State must survive executor failures — hence checkpointing is mandatory for stateful streaming.
Q 45
How do you implement retry logic with exponential backoff for a flaky external API call in a PySpark job?
Answer: B

import time, random
def call_with_retry(func, max_retries=3, base_delay=1.0):
  for attempt in range(max_retries):
    try:
      return func()
    except Exception as e:
      if attempt == max_retries - 1: raise
      delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
      time.sleep(delay)

# Use in mapPartitions for efficiency (one connection per partition)
def process_partition(rows):
  session = create_session() # reuse connection
  for row in rows:
    result = call_with_retry(lambda: api_call(session, row))
    yield result

Additional patterns:
• Dead-letter queue: Catch permanent failures → write to error table
• Circuit breaker: Stop retrying if failure rate exceeds threshold
• spark.task.maxFailures: Spark-level task retry (for executor crashes, not logic errors)
• Use mapPartitions to initialize connections once per partition, not once per row
Q 46
In the Spark UI, what does a very high "Shuffle Write" but very low "Shuffle Read" in the same stage indicate?
Answer: B

Shuffle anatomy:
• Stage N (map side): Produces shuffle write (partitioned output files)
• Stage N+1 (reduce side): Reads those files as shuffle read

So high shuffle WRITE in stage N → high shuffle READ in stage N+1. They're paired across stages, not within the same stage.

Key Spark UI metrics to watch:
• Shuffle Write/Read: High values → expensive network transfer → consider bucketing, broadcasting
• GC Time / Task Time ratio: >10% GC → memory pressure, tune heap or reduce object creation
• Spill (memory/disk): Executor ran out of memory → increase memory or reduce data per task
• Task duration spread: Wide spread = data skew
• Input size vs Output size: Large reduction = effective filter; no reduction = filter not pushed down
• Executor tab: Check active tasks, failed tasks, GC stats per executor
• SQL tab: Physical plan with actual row counts vs estimates — big discrepancy = stale statistics

Q 48
Events arrive up to 2 hours late in your Kafka stream. How do you handle late data without accumulating infinite state?
Answer: B

from pyspark.sql.functions import window, col

df_stream = spark.readStream.format("kafka")...

result = (df_stream
  .withWatermark("event_time", "2 hours") # Accept up to 2hr late
  .groupBy(
    window(col("event_time"), "10 minutes"), # 10-min windows
    col("user_id")
  )
  .count())

How watermarking works:
• Spark tracks max event_time seen = T
• Watermark threshold = T - 2 hours
• Events with event_time < watermark are DROPPED
• Windows that ended before the watermark are FINALIZED and state is evicted

Trade-off: Higher watermark = accept more late data + larger state. Lower watermark = less late data accepted + smaller state.

For append output mode: Results are only emitted after the window is finalized (watermark passes the window end). For update mode, partial results are emitted each trigger and updated when late data arrives.
Q 49
What is the architectural difference between Spark's Catalyst Optimizer and Tungsten Execution Engine?
Answer: B

They operate at completely different levels and are complementary:

Catalyst Optimizer (logical layer):
• Works on the QUERY PLAN — the "what to compute"
• Applies logical transformations: push filters down, eliminate redundant projections, reorder joins, fold constants
• Cost-based optimization: uses table statistics to choose best join strategies
• Language: Trees of plan nodes (works on all languages equally)
• Output: Optimized physical plan

Tungsten Execution Engine (physical layer):
• Works on HOW to execute the physical plan — "how to actually run it"
• Off-heap memory: Manages binary data in native memory, bypasses JVM GC entirely
• Cache-aware algorithms: Data structures optimized for CPU cache lines
• Whole-stage code generation: Compiles entire pipeline into a single optimized JVM function (eliminates virtual function calls, enables SIMD-like operations)
• Binary format: Operates directly on compact binary data without Java object overhead

Pipeline: SQL/DataFrame API → Catalyst (optimize plan) → Tungsten (execute efficiently)
