STEP-BY-STEP EXECUTION
Every phase from code submission to final write
1
STEP 01
SparkSession Creation + Resource Request

Your SparkSession.builder.getOrCreate() launches the Driver JVM on master-01 with 4 GB heap. The Driver contacts the Cluster Manager (YARN) requesting: --num-executors 8 --executor-memory 24g --executor-cores 4.

ComponentMemory AllocatedNote
Driver JVM heap4 GBDAG, metadata, broadcast vars, result collection
Driver overhead1.2 GB10-15% JVM overhead, off-heap buffers
OS / other~10.8 GBRemaining of 16 GB machine RAM
ℹ
At this point ZERO data has been read. Spark is lazy — nothing executes until an action (.count, .show, .write) is called.
2
STEP 02
Executor Launch on All 4 Workers

YARN instructs each Worker Node to start 2 Executor JVMs. Each Executor claims 24 GB heap from the node's 64 GB RAM. Workers reserve 16 GB for OS/HDFS DataNode overhead.

Memory ZoneSize (per Executor)Purpose
Reserved (Spark internal)300 MBspark.memory.fraction excludes this first
Unified Memory Pool (60%)14.2 GBExecution + Storage share dynamically
→ Execution Memory (50%)9.6 GBSorting, hashing, joins, aggregations
→ Storage Memory (50%)9.6 GBCached RDDs/DataFrames, broadcast vars
User Memory (40%)9.48 GBUDFs, user data structures, internal metadata
JVM Overhead~3.6 GB10-15% extra requested from YARN on top of heap

Total cluster execution capacity: 8 executors × 9.6 GB = 76.8 GB execution memory. With 32 slots, each running task gets ~2.4 GB execution memory to work with.

3
STEP 03
DAG Construction (Lazy — No Data Yet)

As you write df.read → .filter → .groupBy → .agg, the Driver's DAG Scheduler builds a logical plan in memory. This is just a graph of operations — no partition is opened.

When you call .write.parquet(...) (an action), the DAG is finalized. The optimizer (Catalyst) rewrites it: pushes filters down, selects scan columns, estimates row counts. The physical plan is compiled to RDD operations.

Logical Plan:
  Write[parquet, /output]
    Aggregate[key → sum(val), count(*)]
      Filter[col > X]
        Scan[parquet, /data/100gb/]

→ After optimization → 2 Stages + 1 Shuffle
→ Stage 1: Scan + Filter (800 tasks)
→ Stage 2: Aggregate + Write (200 tasks)
ℹ
Driver memory used so far: ~200 MB for the DAG, 800 partition descriptors, and task metadata. Very lightweight.
4
STEP 04
Stage 1 — 800 Tasks Dispatched in 25 Waves

The Task Scheduler sends 32 Tasks at a time (one per slot) to Executors. Each Task is serialized (JVM closure) and sent over the network to the Executor. The Task processes one 128 MB partition: opens the HDFS block, decodes Parquet columns, applies the filter, computes the new column.

PhasePer-Task MemoryDetail
Parquet decode buffer128–200 MBRaw partition inflated from compressed Parquet
Filter + project50–80 MBColumnar batch processing (Tungsten/Arrow)
Shuffle write buffer32–64 MBPre-sort + hash partition into 200 buckets
Peak execution mem~400 MBWell within 2.4 GB per task allocation

After each Task finishes it writes shuffle data to local disk on the Worker (External Shuffle Service). 800 Tasks × 200 shuffle buckets = up to 160,000 shuffle files (Spark merges with sort-based shuffle).

✓
Wave 1: Tasks 1–32 run. Wave 2: Tasks 33–64. … Wave 25: Tasks 769–800. After all 25 waves, Stage 1 is complete.
5
STEP 05
Shuffle — Network Data Exchange

This is the most expensive phase. All 200 shuffle partitions must be pulled from all 8 Executors across the network before Stage 2 can begin. Each Stage 2 task fetches its slice from every Stage 1 Executor.

MetricValueNote
Shuffle data size~50–70 GBFiltered/projected data, smaller than 100 GB raw
Shuffle per partition~300 MB70 GB ÷ 200 shuffle partitions
Shuffle fetch buffer~300 MBPer Stage 2 task, held in execution memory
Network pressureHighAll 8 executors simultaneously push/pull data
⚠
If shuffle data doesn't fit in execution memory (9.6 GB per executor), Spark spills to local disk. Spill = slower but no OOM crash. Monitor with Spark UI → Stages → "Shuffle spill (disk)".

The Driver coordinates shuffle metadata (which Executor holds which bucket). Executors use Netty-based HTTP to fetch shuffle blocks from each other's External Shuffle Service.

6
STEP 06
Stage 2 — 200 Tasks: Aggregate + Write

200 Tasks are dispatched (7 waves of 32, final wave = 8 tasks). Each Task reads its shuffle partition (~300 MB of pre-grouped data), applies sum() and count(), and writes one Parquet output file.

PhaseMemory UseDetail
Shuffle fetch buffer~300 MBData fetched from all Stage 1 executors for this key range
Hash aggregation~200 MBHashMap of key → (sum, count) in execution mem
Write buffer~64 MBParquet row group buffer before flushing to S3/HDFS
Peak per task~600 MBComfortable within 2.4 GB execution allocation

Output: 200 Parquet files written to the sink (S3/HDFS), each ~5–10 MB of aggregated results. Total output is dramatically smaller than 100 GB input.

7
STEP 07
Job Complete — Resource Release

All 200 Stage 2 Tasks complete. Driver receives the final commit acknowledgement. The Driver marks the job as SUCCEEDED in the Spark UI and returns control to your application.

If spark.dynamicAllocation.enabled = true, idle Executors are released back to YARN within executorIdleTimeout (default 60s). Their 24 GB × 8 = 192 GB of cluster RAM is freed for other applications.

✓
Total Tasks Executed: 800 + 200 = 1,000
Total Slots Used: 32 concurrent max
Total Waves: ~32 waves across both stages
Peak Cluster Memory Used: ~192 GB execution + storage
